1) Simple Factory Pattern - 

Here is the **clearest explanation of the Simple Factory Pattern** in **JavaScript**, using the same Logistics example (**Truck** and **Ship**).

---

# 🚀 **Simple Factory Pattern (Also called Static Factory)**

⚡ It is *not* a GOF design pattern, but widely used.
⚡ It is the simplest way to centralize object creation.
⚡ Uses **one factory class** with **one method** that returns different objects.

---

# 🟥 Before Simple Factory (Bad Approach)

You directly create objects everywhere using `new`.

```js
const t1 = new Truck();
const t2 = new Ship();
```

### ❌ Problems:

* Repeated code
* If you add `Plane`, you must update everywhere
* Many `new` scattered → hard to maintain

---

# 🟩 Simple Factory Pattern (Good Approach)

A factory method called `createTransport(type)` decides which object to return.

---

# 🟦 **1️⃣ Product Classes**

```js
class Truck {
  deliver() {
    console.log("Delivering cargo by road...");
  }
}

class Ship {
  deliver() {
    console.log("Delivering cargo by sea...");
  }
}
```

---

# 🟦 **2️⃣ Simple Factory**

```js
class TransportFactory {
  static createTransport(type) {
    if (type === "truck") return new Truck();
    if (type === "ship") return new Ship();

    throw new Error("Unknown transport type");
  }
}
```

### ✔ Centralized creation

### ✔ Only one place to maintain

### ✔ Uses `if/else` **inside factory**, not everywhere

---

# 🟦 **3️⃣ Usage**

```js
const t1 = TransportFactory.createTransport("truck");
t1.deliver();
// → Delivering cargo by road...

const t2 = TransportFactory.createTransport("ship");
t2.deliver();
// → Delivering cargo by sea...
```

---

# 🟩 Where Simple Factory is Useful

✔ When you have only **1 factory**
✔ When object creation is simple
✔ When you want to remove scattered `new` keywords
✔ When new product types are rare
✔ Good for small to medium apps

---

# 🟥 Limitations of Simple Factory

Although better than hardcoding:

### ❌ Violates Open–Closed Principle (OCP)

To add a new type:

```js
if (type === "plane") return new Plane();
```

You must modify the factory.

### ❌ Still relies on `if/else`

Better than spreading them everywhere, but still not perfect.

### ❌ Cannot be extended through subclassing

Unlike Factory Method.

---

# 🟩 Simple Factory vs Factory Method vs Abstract Factory

| Pattern              | Why Used?                        | Pros                  | Cons         |
| -------------------- | -------------------------------- | --------------------- | ------------ |
| **Simple Factory**   | Centralize object creation       | Easy, single place    | Breaks OCP   |
| **Factory Method**   | Subclass decides product         | Extensible            | More classes |
| **Abstract Factory** | Create multiple related products | Clean family creation | Most complex |

---

# 🎯 One-Line Summary

**Simple Factory → One class decides which product to create based on a parameter.**

2) Factory Method Pattern -

Great — here is the **BEST AND SIMPLEST Logistics example** of **Factory Method** using **Truck** and **Ship** as products.
I’ll show you:

1️⃣ BEFORE Factory Method (bad way)
2️⃣ AFTER Factory Method (correct way)
3️⃣ Clear explanation

---

# 🚫 **BEFORE Factory Method**

(You manually create Truck/Ship using if-else)

You have a Logistics system that delivers cargo using **Truck** or **Ship**.

## ❌ The Problem

* Logic is coupled with `Truck` and `Ship`
* Adding `Airplane` means modifying multiple places
* Violates Open–Closed Principle

---

## ❌ **Before Example Code**

```js
class Logistics {
  deliver(type) {
    let transport;

    if (type === "truck") {
      transport = new Truck();
    } else if (type === "ship") {
      transport = new Ship();
    }

    transport.start();
    transport.deliverCargo();
  }
}

class Truck {
  start() {
    console.log("Truck starting to drive...");
  }
  deliverCargo() {
    console.log("Delivering cargo by road");
  }
}

class Ship {
  start() {
    console.log("Ship starting to sail...");
  }
  deliverCargo() {
    console.log("Delivering cargo by sea");
  }
}

const logistics = new Logistics();
logistics.deliver("truck");
```

---

## ❌ **Issues**

* Adding `Plane` → must add new `if`
* Code becomes long and messy
* Parent class knows too much about object creation
* Hard to maintain

---

# ✅ **AFTER Factory Method**

Parent class defines the **process**.
Child class decides **which transport object** to create.

---

# 🟩 **1️⃣ Product Classes (Truck, Ship)**

```js
class Truck {
  start() {
    console.log("Truck driving on the road...");
  }
  deliverCargo() {
    console.log("Cargo delivered by Truck.");
  }
}

class Ship {
  start() {
    console.log("Ship sailing on the sea...");
  }
  deliverCargo() {
    console.log("Cargo delivered by Ship.");
  }
}
```

---

# 🟩 **2️⃣ Superclass Defines the Factory Method**

```js
class Logistics {
  // Factory Method
  createTransport() {
    throw new Error("Subclass must implement createTransport()");
  }

  // Business logic (same for all transports)
  planDelivery() {
    const transport = this.createTransport();
    transport.start();
    transport.deliverCargo();
  }
}
```

✔ Parent knows **HOW to deliver**,
❌ but does **NOT** know **WHICH** transport.

---

# 🟩 **3️⃣ Subclasses Decide the Transport Type**

```js
class RoadLogistics extends Logistics {
  createTransport() {
    return new Truck();
  }
}

class SeaLogistics extends Logistics {
  createTransport() {
    return new Ship();
  }
}
```

✔ Each subclass provides the correct transport
✔ No need for `if / else`

---

# 🟩 **4️⃣ Usage**

```js
const road = new RoadLogistics();
road.planDelivery();
// → Truck driving on the road...
// → Cargo delivered by Truck.

const sea = new SeaLogistics();
sea.planDelivery();
// → Ship sailing on the sea...
// → Cargo delivered by Ship.
```

---

# 🟥 BEFORE vs AFTER (Quick Comparison)

| Feature                      | BEFORE                 | AFTER (Factory Method)  |
| ---------------------------- | ---------------------- | ----------------------- |
| Uses `new Truck()` in parent | ❌ Yes                  | ✔ No                    |
| Uses `if / else`             | ❌ Yes                  | ✔ No                    |
| Add new product (Airplane)   | ❌ Modify existing code | ✔ Add new subclass only |
| Follows Open–Closed?         | ❌ No                   | ✔ Yes                   |
| Who decides object creation? | Parent                 | Child subclass          |

---

# 🎯 **One Line Summary**

**Factory Method moves object creation logic from the parent to its subclasses.**

3) Abstract Factory Pattern -

Below is a **deep, structured explanation of the Abstract Factory Pattern**, tied to what you already know about **Simple Factory** and **Factory Method**, and using **clear JavaScript examples**.

---

# 🏭 Abstract Factory Pattern — In Depth

## 1️⃣ What Is Abstract Factory? (Core Idea)

> **Abstract Factory provides an interface for creating *families of related objects* without specifying their concrete classes.**

### In simple words:

* You don’t create **one object**
* You create **a set (family) of objects** that are designed to work together
* You switch the **entire family** by switching the factory

---

## 2️⃣ Why Do We Need Abstract Factory?

Factory Method creates **one product**.

But in real systems, objects come in **groups**:

### Logistics Example

When you choose a logistics mode, you don’t just choose:

* 🚚 Truck
  You also need:
* 👨‍✈️ Driver
* 📄 Route Planner
* 🧾 Insurance Policy

These must be **compatible** with each other.

Abstract Factory ensures:

* Road logistics objects work together
* Sea logistics objects work together
* No mixing (Truck + Ship Captain ❌)

---

## 3️⃣ Structure of Abstract Factory

### Participants

| Role                | Description                                |
| ------------------- | ------------------------------------------ |
| **AbstractFactory** | Declares methods for creating each product |
| **ConcreteFactory** | Implements methods for a specific family   |
| **AbstractProduct** | Interface/base class for products          |
| **ConcreteProduct** | Actual implementations                     |
| **Client**          | Uses only abstract interfaces              |

---

## 4️⃣ Logistics Example (Truck / Ship family)

### Families:

* **Road family** → Truck + RoadDriver
* **Sea family** → Ship + SeaCaptain

---

## 5️⃣ Step-by-Step JavaScript Implementation

---

### 🟦 1. Abstract Products

```js
class Transport {
  deliver() {
    throw new Error("deliver() must be implemented");
  }
}

class Driver {
  drive() {
    throw new Error("drive() must be implemented");
  }
}
```

---

### 🟦 2. Concrete Products

```js
// Road products
class Truck extends Transport {
  deliver() {
    console.log("Delivering cargo by road");
  }
}

class RoadDriver extends Driver {
  drive() {
    console.log("Driving a truck");
  }
}

// Sea products
class Ship extends Transport {
  deliver() {
    console.log("Delivering cargo by sea");
  }
}

class SeaCaptain extends Driver {
  drive() {
    console.log("Sailing a ship");
  }
}
```

---

### 🟦 3. Abstract Factory

```js
class LogisticsFactory {
  createTransport() {
    throw new Error("createTransport() must be implemented");
  }

  createDriver() {
    throw new Error("createDriver() must be implemented");
  }
}
```

☝️ This factory defines **what products exist in a family**.

---

### 🟦 4. Concrete Factories

```js
class RoadLogisticsFactory extends LogisticsFactory {
  createTransport() {
    return new Truck();
  }

  createDriver() {
    return new RoadDriver();
  }
}

class SeaLogisticsFactory extends LogisticsFactory {
  createTransport() {
    return new Ship();
  }

  createDriver() {
    return new SeaCaptain();
  }
}
```

Each factory guarantees **compatible objects**.

---

### 🟦 5. Client Code (Important Part)

```js
function startDelivery(factory) {
  const transport = factory.createTransport();
  const driver = factory.createDriver();

  driver.drive();
  transport.deliver();
}
```

☝️ Client does NOT know:

* Truck or Ship
* Road or Sea
* Concrete class names

---

### 🟦 6. Usage

```js
const roadFactory = new RoadLogisticsFactory();
startDelivery(roadFactory);

const seaFactory = new SeaLogisticsFactory();
startDelivery(seaFactory);
```

### Output

```
Driving a truck
Delivering cargo by road

Sailing a ship
Delivering cargo by sea
```

---

## 6️⃣ Key Benefits (Why It Exists)

### ✅ Ensures compatibility

* No mixing families accidentally

### ✅ Single switch point

* Change entire system by switching factory

### ✅ Strong decoupling

* Client depends on abstractions only

### ✅ Open–Closed Principle

* Add new family → new factory
* No client code change

---

## 7️⃣ Comparison with Factory Method

| Aspect           | Factory Method | Abstract Factory   |
| ---------------- | -------------- | ------------------ |
| Creates          | One product    | Family of products |
| Uses inheritance | Yes            | Yes                |
| Switch behavior  | Subclass       | Factory object     |
| Example          | Truck OR Ship  | Truck + Driver     |
| Complexity       | Medium         | Higher             |

---

## 8️⃣ Common Real-World Uses

| Domain              | Abstract Factory Usage               |
| ------------------- | ------------------------------------ |
| UI Frameworks       | LightThemeFactory / DarkThemeFactory |
| Cross-platform apps | WindowsFactory / MacFactory          |
| Databases           | MySQLFactory / MongoFactory          |
| Cloud SDKs          | AWSFactory / AzureFactory            |
| Testing             | RealFactory / MockFactory            |

---

## 9️⃣ When NOT to Use Abstract Factory

❌ If you only need one object
❌ If product families are unlikely to grow
❌ If complexity outweighs benefits

Use **Factory Method** instead.

---

## 🔑 Final Mental Model

> **Factory Method**
> → Chooses *how to create one object*

> **Abstract Factory**
> → Chooses *which family of objects to use*

---

## 🧠 One-Line Summary

**Abstract Factory lets you swap entire product families without changing client code.**

4) Builder Pattern -
Below is an **in-depth, end-to-end explanation of the Builder Pattern**, with **concepts, structure, variations, pros/cons, and real-world + JavaScript examples**—the level expected in **system design / solution architect interviews**.

---

# 🧱 Builder Pattern (Creational Design Pattern)

## 1️⃣ What is the Builder Pattern?

**Definition (GoF)**

> Builder Pattern separates the **construction of a complex object** from its **representation**, so the same construction process can create different representations.

In simple terms:

* You **build an object step by step**
* Construction logic is **decoupled** from the final object
* Especially useful when an object has **many optional parameters**

---

## 2️⃣ Problem It Solves (Why Builder?)

### ❌ Problem: Telescoping Constructors

```js
class House {
  constructor(
    walls,
    doors,
    windows,
    hasGarage,
    hasSwimmingPool,
    hasGarden
  ) {
    this.walls = walls;
    this.doors = doors;
    this.windows = windows;
    this.hasGarage = hasGarage;
    this.hasSwimmingPool = hasSwimmingPool;
    this.hasGarden = hasGarden;
  }
}
```

Usage becomes ugly:

```js
new House(4, 2, 6, true, false, true);
```

❗ Problems:

* Hard to read
* Easy to make mistakes
* Constructor explodes as options grow

---

## 3️⃣ Builder Pattern – Core Idea

Instead of passing everything at once:

* Use a **Builder** to configure step by step
* Call `.build()` at the end

```js
new HouseBuilder()
  .setWalls(4)
  .setDoors(2)
  .setWindows(6)
  .setGarage(true)
  .build();
```

✔ Readable
✔ Flexible
✔ Safe

---

## 4️⃣ Structure (UML-level Understanding)

### Participants

| Component               | Responsibility              |
| ----------------------- | --------------------------- |
| **Product**             | Final complex object        |
| **Builder (interface)** | Declares building steps     |
| **ConcreteBuilder**     | Implements steps            |
| **Director (optional)** | Orchestrates building order |

---

## 5️⃣ Simple Builder (Fluent Builder – Most Common in JS)

### Product

```js
class House {
  constructor(builder) {
    this.walls = builder.walls;
    this.doors = builder.doors;
    this.windows = builder.windows;
    this.garage = builder.garage;
    this.pool = builder.pool;
  }
}
```

### Builder

```js
class HouseBuilder {
  setWalls(walls) {
    this.walls = walls;
    return this;
  }

  setDoors(doors) {
    this.doors = doors;
    return this;
  }

  setWindows(windows) {
    this.windows = windows;
    return this;
  }

  setGarage(hasGarage) {
    this.garage = hasGarage;
    return this;
  }

  setPool(hasPool) {
    this.pool = hasPool;
    return this;
  }

  build() {
    return new House(this);
  }
}
```

### Usage

```js
const house = new HouseBuilder()
  .setWalls(4)
  .setDoors(2)
  .setGarage(true)
  .build();
```

---

## 6️⃣ Builder with Director (Classic GoF Style)

### When do we need Director?

* When **construction steps are fixed**
* But **representation varies**

### Director

```js
class HouseDirector {
  constructor(builder) {
    this.builder = builder;
  }

  buildLuxuryHouse() {
    return this.builder
      .setWalls(4)
      .setDoors(4)
      .setWindows(10)
      .setGarage(true)
      .setPool(true)
      .build();
  }

  buildSimpleHouse() {
    return this.builder
      .setWalls(4)
      .setDoors(1)
      .setWindows(4)
      .build();
  }
}
```

### Usage

```js
const builder = new HouseBuilder();
const director = new HouseDirector(builder);

const luxuryHouse = director.buildLuxuryHouse();
```

---

## 7️⃣ Real-World Analogy

### 🍔 Burger Builder

```js
new BurgerBuilder()
  .addBun("wheat")
  .addPatty("veg")
  .addCheese()
  .addSauce("mayo")
  .build();
```

This is **exactly how Builder Pattern works**.

---

## 8️⃣ Immutability-Friendly Builder

Common in **Java, Kotlin, system-level code**:

```js
class User {
  constructor({ name, age, email }) {
    this.name = name;
    this.age = age;
    this.email = email;
    Object.freeze(this);
  }
}

class UserBuilder {
  setName(name) {
    this.name = name;
    return this;
  }

  setAge(age) {
    this.age = age;
    return this;
  }

  setEmail(email) {
    this.email = email;
    return this;
  }

  build() {
    return new User(this);
  }
}
```

---

## 9️⃣ Builder vs Factory (Very Important)

| Aspect                   | Builder          | Factory                    |
| ------------------------ | ---------------- | -------------------------- |
| Object creation          | Step-by-step     | One-shot                   |
| Handles complexity       | ✔ Very good      | ❌ Limited                  |
| Optional params          | ✔ Excellent      | ❌ Hard                     |
| Multiple representations | ✔ Yes            | ✔ Yes                      |
| Focus                    | **How to build** | **Which object to create** |

👉 **Factory = selection logic**
👉 **Builder = construction logic**

---

## 🔟 When Should You Use Builder Pattern?

### ✅ Use Builder When:

* Object has **many optional fields**
* Construction is **complex**
* Need **readable & maintainable creation**
* Want **immutability**
* Same construction process → different objects

### ❌ Avoid When:

* Object is simple
* Few parameters
* No construction complexity

---

## 1️⃣1️⃣ Common Interview Traps ⚠️

❌ Confusing Builder with Factory
❌ Overusing Builder for simple objects
❌ Forgetting `.build()` validation

### Validation Example

```js
build() {
  if (!this.walls || !this.doors) {
    throw new Error("Walls and doors are mandatory");
  }
  return new House(this);
}
```

---

## 1️⃣2️⃣ Builder Pattern Variants

1. **Fluent Builder** (most common in JS)
2. **Classic GoF Builder (with Director)**
3. **Step Builder** (forces build order)
4. **Immutable Builder**
5. **Nested Builder** (for very complex objects)

---

## 1️⃣3️⃣ Step Builder (Advanced)

Enforces order at compile/design level (mostly Java/TS)

```js

  .setWalls(4)
  .setDoors(2)
  .setWindows(6)
  .build();
```

Prevents skipping required steps.

* **React** → JSX is a builder
* **Axios** → request config builder
* **AWS SDK** → request builders
* **Java Lombok @Builder**
* **UI Form builders**

---

> “Builder Pattern is used to construct complex objects step by step, separating construction logic from representation, especially when there are many optional parameters.”

Question - in build method, we have "return new Home(this);" even if we just used "return this". we got same object. then why we should go for "return new Home(this);"

Excellent question — this is a **very important Builder-pattern nuance**, and interviewers *love* this one.

Short answer:
👉 **`return this` returns the *builder*, not the *product*.**
👉 **`return new Home(this)` returns the *final object* and freezes the construction phase.**

Let’s go step by step, deeply but clearly.

---

## 1️⃣ What does `this` actually mean inside `build()`?

```js
class HomeBuilder {
  setRooms(r) {
    this.rooms = r;
    return this;
  }

  build() {
    return this;   // ❌
  }
}
```

Here:

* `this` → **HomeBuilder instance**
* NOT a Home
* You are returning the **builder**, not the **product**

So this is what you get:

```js
const home = new HomeBuilder().setRooms(3).build();
```

`home` is still a **HomeBuilder**, not a **Home**.

---

## 2️⃣ Why is that a problem?

### ❌ Builder ≠ Product

| Builder            | Product (Home)     |
| ------------------ | ------------------ |
| Temporary          | Final              |
| Mutable            | Often immutable    |
| Has setter methods | Has domain methods |
| Construction logic | Business logic     |

Returning `this` **leaks construction details**.

---

## 3️⃣ Correct Approach: Create a Separate Product

```js
class Home {
  constructor(builder) {
    this.rooms = builder.rooms;
    this.garage = builder.garage;
    Object.freeze(this); // optional but common
  }
}
```

```js
class HomeBuilder {
  build() {
    return new Home(this); // ✅
  }
}
```

Now:

```js
const home = new HomeBuilder()
  .setRooms(3)
  .setGarage(true)
  .build();
```

✔ `home` is a **Home**
✔ Builder is discarded
✔ Construction phase ends

---

## 4️⃣ Key Concept: **Separation of Concerns**

### With `return this` ❌

* Builder is used as Product
* Construction + Representation mixed
* Violates **Single Responsibility Principle**

### With `return new Home(this)` ✅

* Builder → builds
* Home → represents
* Clean design

---

## 5️⃣ Immutability (Very Important Reason)

### If you return `this`

```js
const home = builder.build();
builder.setRooms(10);

console.log(home.rooms); // 😱 changes!
```

Because `home === builder`

---

### If you return a new object

```js
const home = builder.build();
builder.setRooms(10);

console.log(home.rooms); // ✅ unchanged
```

✔ Product is **stable**
✔ Builder can be reused or discarded

---

## 6️⃣ Validation Happens at `build()`

Builder is the **gatekeeper**.

```js
build() {
  if (!this.rooms) {
    throw new Error("Rooms required");
  }
  return new Home(this);
}
```

If you return `this`, you **can’t enforce validation** cleanly.

---

## 7️⃣ Reuse Builder for Multiple Products

```js
const builder = new HomeBuilder().setRooms(3);

const home1 = builder.build();
const home2 = builder.setGarage(true).build();
```

Works **only** if build creates a **new product each time**.

---

## 8️⃣ When *is* `return this` acceptable?

Only in **non-Builder APIs** like config objects:

```js
axios
  .setHeader(...)
  .setTimeout(...)
  .send();
```

But that’s **not a GoF Builder Pattern** — it’s a **fluent configuration object**.

---

## 9️⃣ Interview-Grade Answer (Memorize This)

> We return `new Home(this)` instead of `this` because the Builder is a temporary, mutable construction object, while `Home` is the final, immutable product. Returning the builder leaks construction details, breaks immutability, and violates separation of concerns.

5) Adapter Pattern 

Let’s go **deep and practical** on the **Adapter Pattern**, with **why, when, structure, variants, pitfalls, and real-world + JS examples**. Since you’re learning design patterns seriously, I’ll treat this like **architect-level understanding**, not just definition.

---

# Adapter Pattern (In Depth)

## 1. What problem does Adapter solve?

> **Adapter allows incompatible interfaces to work together without changing their existing code.**

You already have:

* **Existing / legacy code** (stable, tested, cannot change)
* **New code** that expects a **different interface**

👉 Adapter sits **in between** and **translates** one interface into another.

---

## 2. Real-world analogy (very important)

### Mobile charger adapter 🔌

* Wall socket → **220V Indian socket**
* Charger → **expects a specific plug**

You **don’t change the socket**
You **don’t change the charger**

👉 You add an **adapter**

This is exactly what the Adapter Pattern does in software.

---

## 3. When should you use Adapter?

Use Adapter when:

✅ You want to reuse an **existing class**
✅ But its interface **does not match** what your system expects
✅ You **cannot or should not modify** the existing class
✅ You want to follow **Open–Closed Principle**

🚫 Do NOT use Adapter if:

* You can directly modify the class
* Interfaces already match
* You’re just mapping DTOs (that’s not Adapter)

---

## 4. Structure (Participants)

```
Client  --->  Target (interface)
                  ↑
              Adapter
                  ↓
              Adaptee
```

### Roles:

| Role        | Meaning                                    |
| ----------- | ------------------------------------------ |
| **Client**  | Code that needs a specific interface       |
| **Target**  | Interface expected by the client           |
| **Adaptee** | Existing class with incompatible interface |
| **Adapter** | Converts Adaptee to Target                 |

---

## 5. Simple Example (Conceptual)

### Target (what client expects)

```js
class PaymentProcessor {
  pay(amount) {
    throw new Error("Not implemented");
  }
}
```

### Adaptee (existing / legacy class)

```js
class OldPaymentGateway {
  makePayment(valueInRupees) {
    console.log(`Paid ₹${valueInRupees}`);
  }
}
```

### Adapter

```js
class PaymentAdapter extends PaymentProcessor {
  constructor(oldGateway) {
    super();
    this.oldGateway = oldGateway;
  }

  pay(amount) {
    this.oldGateway.makePayment(amount);
  }
}
```

### Client

```js
const oldGateway = new OldPaymentGateway();
const payment = new PaymentAdapter(oldGateway);

payment.pay(500);
```

✔ Client never knows about `OldPaymentGateway`

---

## 6. Adapter vs Direct Wrapping (important distinction)

❌ This is NOT Adapter:

```js
oldGateway.makePayment(500);
```

✔ This IS Adapter:

```js
payment.pay(500);
```

Why?

* Adapter **implements Target interface**
* Client depends on **abstraction**, not concrete class

---

## 7. Two Types of Adapter Pattern

### 1️⃣ Object Adapter (Most Common ✅)

Uses **composition**

```js
class Adapter {
  constructor(adaptee) {
    this.adaptee = adaptee;
  }
}
```

✔ Flexible
✔ Works in JS, Java, TS
✔ Preferred approach

---

### 2️⃣ Class Adapter (Inheritance-based)

Uses **inheritance** (possible in Java, C++, not ideal in JS)

```java
class Adapter extends Adaptee implements Target
```

🚫 Less flexible
🚫 Tightly coupled
🚫 Rarely used in modern systems

---

## 8. Real-World System Example (API Integration)

### Scenario

Your system expects:

```js
getUser(id)
```

Third-party API gives:

```js
fetchUserByUUID(uuid)
```

---

### Target

```js
class UserService {
  getUser(id) {}
}
```

### Adaptee

```js
class ExternalUserAPI {
  fetchUserByUUID(uuid) {
    return { uuid, name: "Sumeet" };
  }
}
```

### Adapter

```js
class UserAdapter extends UserService {
  constructor(api) {
    super();
    this.api = api;
  }

  getUser(id) {
    return this.api.fetchUserByUUID(id);
  }
}
```

### Client

```js
const service = new UserAdapter(new ExternalUserAPI());
service.getUser("123");
```

---

## 9. Adapter vs Related Patterns (VERY IMPORTANT)

| Pattern       | Difference                                |
| ------------- | ----------------------------------------- |
| **Adapter**   | Makes incompatible interfaces work        |
| **Facade**    | Simplifies a complex system               |
| **Decorator** | Adds behavior without changing interface  |
| **Proxy**     | Controls access                           |
| **Bridge**    | Separates abstraction from implementation |

👉 **Adapter = interface translation**

---

## 10. Adapter + SOLID Principles

✔ **Single Responsibility**

* Adapter only translates interface

✔ **Open–Closed**

* No modification of existing classes

✔ **Dependency Inversion**

* Client depends on Target interface

---

## 11. Common Mistakes

❌ Treating simple helper functions as Adapter
❌ Putting business logic inside Adapter
❌ Adapter calling multiple unrelated classes
❌ Overusing Adapter for internal code

---

## 12. When Adapter becomes a code smell 🚨

* Too many adapters → bad initial design
* Adapter doing transformations + validation + logic
* Adapters chained deeply

👉 That’s when **Facade or refactoring** is better.

---

## 13. Adapter in Frontend / Node.js (Practical)

### Example: Logger Adapter

Different log libraries:

```js
winston.log()
pino.info()
```

You want:

```js
logger.log("message")
```

👉 Create adapters for each logger.

---

## 14. One-line Summary (Interview Gold)

> **Adapter Pattern converts the interface of a class into another interface the client expects, allowing incompatible classes to work together without modifying their source code.**

6) Bridge Pattern -

Below is a **deep, practical explanation of the Bridge Design Pattern**, tailored for **system design & front-end / JS understanding**, since you’re learning patterns in depth.

---

# Bridge Design Pattern — In Depth

## 1. Intent (Why Bridge Exists)

> **Bridge is a structural design pattern that splits a large class or closely related classes into two separate hierarchies:**

* **Abstraction**
* **Implementation**

These two hierarchies can **evolve independently**.

### Core Problem It Solves

👉 **Class explosion caused by inheritance**

---

## 2. The Core Idea (One Line)

> **“Prefer composition over inheritance by separating WHAT an object does from HOW it does it.”**

---

## 3. When You Need Bridge (Red Flags 🚩)

You should think of Bridge when:

* You have **multiple dimensions of variation**
* Changes in one dimension should **not affect** the other
* Inheritance leads to:

  * Many subclasses
  * Rigid design
  * Difficult maintenance

---

## 4. Classic Example Without Bridge (Problem)

### Example: Notifications

### Variations:

1. **Type of notification**

   * Alert
   * Reminder
2. **Channel**

   * Email
   * SMS
   * Push

### Without Bridge (Inheritance Explosion ❌)

```
AlertEmail
AlertSMS
AlertPush
ReminderEmail
ReminderSMS
ReminderPush
```

👉 Every new notification type or channel **multiplies classes**

---

## 5. Bridge Pattern Structure

### Key Participants

```
Abstraction  ────────▶ Implementor
     ▲                     ▲
RefinedAbstraction   ConcreteImplementor
```

### Breakdown

| Role                    | Responsibility           |
| ----------------------- | ------------------------ |
| **Abstraction**         | High-level control logic |
| **Implementor**         | Low-level operations     |
| **ConcreteImplementor** | Actual implementation    |
| **RefinedAbstraction**  | Variants of abstraction  |

---

## 6. Bridge Pattern Applied (Correct Design ✅)

### Step 1: Implementor Interface (Channel)

```js
class NotificationSender {
  send(message) {
    throw new Error("Method not implemented");
  }
}
```

---

### Step 2: Concrete Implementations

```js
class EmailSender extends NotificationSender {
  send(message) {
    console.log("Email sent:", message);
  }
}

class SMSSender extends NotificationSender {
  send(message) {
    console.log("SMS sent:", message);
  }
}
```

---

### Step 3: Abstraction

```js
class Notification {
  constructor(sender) {
    this.sender = sender; // BRIDGE
  }

  notify(message) {
    this.sender.send(message);
  }
}
```

---

### Step 4: Refined Abstractions

```js
class AlertNotification extends Notification {
  notify(message) {
    this.sender.send("ALERT: " + message);
  }
}

class ReminderNotification extends Notification {
  notify(message) {
    this.sender.send("REMINDER: " + message);
  }
}
```

---

### Step 5: Usage

```js
const email = new EmailSender();
const sms = new SMSSender();

const alertEmail = new AlertNotification(email);
alertEmail.notify("Server down");

const reminderSMS = new ReminderNotification(sms);
reminderSMS.notify("Pay credit card bill");
```

---

## 7. What Changed (Key Benefits)

| Without Bridge    | With Bridge           |
| ----------------- | --------------------- |
| Class explosion   | Small class hierarchy |
| Tight coupling    | Loose coupling        |
| Hard to extend    | Easy to extend        |
| Inheritance-heavy | Composition-based     |

👉 **Add a new channel?**

* Just create a new sender
* No notification classes touched

👉 **Add a new notification type?**

* Just extend `Notification`
* No sender classes touched

---

## 8. Bridge vs Adapter (Important Interview Question ⚠️)

| Bridge                                            | Adapter                       |
| ------------------------------------------------- | ----------------------------- |
| Designed **upfront**                              | Used **after** code exists    |
| Prevents class explosion                          | Fixes incompatible interfaces |
| Abstraction & implementation evolve independently | Makes old code work with new  |

**Memory trick:**

* 🧱 Bridge = architecture decision
* 🔌 Adapter = compatibility fix

---

## 9. Bridge vs Strategy (Common Confusion)

| Bridge                               | Strategy                   |
| ------------------------------------ | -------------------------- |
| Structural pattern                   | Behavioral pattern         |
| Focus: abstraction vs implementation | Focus: algorithm switching |
| Long-term design                     | Runtime behavior choice    |

👉 Bridge is about **structure**
👉 Strategy is about **behavior**

---

## 10. Real-World Examples 🌍

### Frontend

* UI Components + Rendering engines
* Theme system + UI controls

### Backend

* Database abstraction (ORM)
* Payment processing (Stripe, Razorpay, PayPal)

### System Design

* Message queue abstraction (Kafka / SQS / RabbitMQ)
* Storage abstraction (S3 / GCS / Azure Blob)

---

## 11. When NOT to Use Bridge ❌

* Only **one dimension of variation**
* System is small & unlikely to grow
* You want quick & simple code

Overusing Bridge leads to **unnecessary complexity**

---

## 12. Interview-Ready Definition 🎯

> **Bridge pattern decouples an abstraction from its implementation so that both can vary independently, avoiding class explosion caused by inheritance.**

---

## 13. Mental Model (Very Important)

> **Bridge = Two hierarchies connected by composition**

```
Business Logic  ───▶ Technical Detail
   (what)             (how)
```

If you use inheritance:

```
SonyBasicRemote
```

```js
class TV {
```

```js
class SonyTV extends TV {
```

## One-Line Understanding 💡

> **Bridge = Remote (what) + TV (how), connected by composition**

## Why This Is Powerful

* Add new TV brand ➜ no remote changes

## Ultra-short version (Interview)

> **Bridge pattern separates abstraction (Remote) from implementation (TV) so both can change independently.**

7) Composite Pattern - 

Let’s go **deep but simple** with the **Composite Design Pattern**, keeping examples intuitive and practical (the way you’ve been learning Adapter, Builder, Bridge, etc.).

---

# Composite Design Pattern (In Depth)

## 1️⃣ What problem does Composite solve?

**Problem:**
You want to treat **individual objects** and **groups of objects** in the **same way**.

> “Whether it’s one thing or a collection of things, I want to call the same method.”

### Real-life analogy

* A **file** and a **folder**
* A **folder** can contain:

  * files
  * subfolders (which again contain files/folders)

You don’t want different code paths like:

```js
if (isFile) { ... }
if (isFolder) { ... }
```

👉 **Composite lets you handle both uniformly.**

---

## 2️⃣ Intent (Official Definition – simplified)

> Composite pattern lets you compose objects into tree structures and work with them as if they were individual objects.

Key words:

* **Tree structure**
* **Part–whole relationship**
* **Same interface for single & group**

---

## 3️⃣ Structure of Composite Pattern

There are **3 main roles**:

```
Component (interface / abstract class)
   ↑
 ┌───────────────┐
 │               │
Leaf           Composite
```

### 1. Component

* Common interface
* Declares operations like `render()`, `price()`, `execute()`

### 2. Leaf

* Represents **individual object**
* Implements Component
* Has no children

### 3. Composite

* Represents **group of objects**
* Implements Component
* Holds children (Component type)

---

## 4️⃣ Simple Example (Very Important)

### 🎯 Example: **Company Organization**

* Employee → **Leaf**
* Manager → **Composite**
* Both have `getSalary()`

---

### Step 1: Component

```js
class Employee {
  getSalary() {
    throw new Error("Method not implemented");
  }
}
```

---

### Step 2: Leaf (Individual employee)

```js
class Developer extends Employee {
  constructor(name, salary) {
    super();
    this.name = name;
    this.salary = salary;
  }

  getSalary() {
    return this.salary;
  }
}
```

---

### Step 3: Composite (Manager with subordinates)

```js
class Manager extends Employee {
  constructor(name) {
    super();
    this.name = name;
    this.employees = [];
  }

  add(employee) {
    this.employees.push(employee);
  }

  getSalary() {
    return this.employees.reduce(
      (total, emp) => total + emp.getSalary(),
      0
    );
  }
}
```

---

### Step 4: Client Code (Key Benefit)

```js
const dev1 = new Developer("A", 50000);
const dev2 = new Developer("B", 60000);

const manager = new Manager("C");
manager.add(dev1);
manager.add(dev2);

console.log(manager.getSalary()); // 110000
```

✅ **Client doesn’t care** if it’s a Developer or Manager
✅ Same method → `getSalary()`

---

## 5️⃣ Tree Structure Visualization

```
Manager
 ├── Developer A (50k)
 └── Developer B (60k)
```

Recursive calls:

```
manager.getSalary()
 → dev1.getSalary()
 → dev2.getSalary()
```

---

## 6️⃣ Key Characteristics

| Aspect       | Explanation                  |
| ------------ | ---------------------------- |
| Structure    | Tree                         |
| Relationship | Part–Whole                   |
| Interface    | Same for leaf & composite    |
| Behavior     | Recursive delegation         |
| Client       | Treats all objects uniformly |

---

## 7️⃣ When to Use Composite Pattern

✅ Use it when:

* You have **hierarchical / tree-like data**
* You want to **ignore differences** between single and group objects
* Operations need to be applied **recursively**

### Common real-world use cases

* File system (file / folder)
* UI components (button / panel / window)
* Organization structure
* Menu systems
* HTML DOM
* Permission structures

---

## 8️⃣ When NOT to Use

❌ Avoid if:

* You don’t have a hierarchy
* Leaf and Composite behavior is very different
* You need strict control over child operations

---

## 9️⃣ Advantages

✔ Simplifies client code
✔ Open/Closed Principle (add new types easily)
✔ Natural recursion handling
✔ Clean tree modeling

---

## 🔟 Disadvantages

❌ Hard to restrict what can be added as child
❌ Can become **too generic**
❌ Debugging deep trees can be tricky

---

## 1️⃣1️⃣ Composite vs Related Patterns

| Pattern   | Difference                         |
| --------- | ---------------------------------- |
| Composite | Treat individual & group uniformly |
| Decorator | Adds behavior dynamically          |
| Iterator  | Traverses collection               |
| Flyweight | Shares object data                 |

👉 Composite focuses on **structure**, not behavior extension.

---

## 1️⃣2️⃣ UI Example (Very Common)

```js
class UIComponent {
  render() {}
}

class Button extends UIComponent {
  render() {
    console.log("Render Button");
  }
}

class Panel extends UIComponent {
  constructor() {
    super();
    this.children = [];
  }

  add(component) {
    this.children.push(component);
  }

  render() {
    this.children.forEach(child => child.render());
  }
}
```

Usage:

```js
const panel = new Panel();
panel.add(new Button());
panel.add(new Button());

panel.render();
```

---

## 1️⃣3️⃣ One-Line Summary (Interview Gold)

> **Composite lets you treat a single object and a group of objects in the same way by organizing them in a tree structure.**

8) Facade Pattern -

Below is an **in-depth explanation of the Facade Design Pattern**, aligned with your ongoing **design-patterns learning track** and explained in a **system-design friendly way** (how architects actually think about it).

---

## 1. What is the Facade Pattern?

**Facade Pattern** is a **structural design pattern** that provides a **simplified, unified interface** to a **complex subsystem**.

> 👉 Instead of the client talking to many complex classes, it talks to **one Facade class**.

**Intent (GoF):**

> “Provide a unified interface to a set of interfaces in a subsystem.”

---

## 2. The Core Problem It Solves

### Without Facade ❌

* Client knows **too much** about internal classes
* High **coupling**
* Hard to:

  * change subsystem
  * reuse code
  * onboard new developers

### With Facade ✅

* Client talks to **one entry point**
* Subsystem complexity is hidden
* Looser coupling
* Cleaner APIs

---

## 3. Real-World Analogy 🏨

### Hotel Reception Desk

* You **don’t call** housekeeping, kitchen, billing, maintenance separately
* You call **reception**
* Reception coordinates everything internally

➡️ **Reception = Facade**
➡️ **Hotel departments = Subsystem**

---

## 4. Structure (Conceptual)

```
Client
   |
   v
Facade
   |
--------------------------------
|        |          |          |
SubsystemA  SubsystemB  SubsystemC
```

* Client → Facade
* Facade → Subsystem classes
* Client **does not know** subsystem details

---

## 5. Simple Example (Before Facade)

### Problem: Online Order Processing

Client directly uses multiple systems:

```js
payment.processPayment();
inventory.checkStock();
shipping.createShipment();
notification.sendEmail();
```

### Issues

* Client tightly coupled to **4 systems**
* Any change → client code changes
* Hard to reuse

---

## 6. Applying Facade (After)

### Subsystems (unchanged)

```js
class PaymentService {
  processPayment() {
    console.log("Payment processed");
  }
}

class InventoryService {
  checkStock() {
    console.log("Stock available");
  }
}

class ShippingService {
  createShipment() {
    console.log("Shipment created");
  }
}

class NotificationService {
  sendEmail() {
    console.log("Email sent");
  }
}
```

---

### Facade Class ✅

```js
class OrderFacade {
  constructor() {
    this.payment = new PaymentService();
    this.inventory = new InventoryService();
    this.shipping = new ShippingService();
    this.notification = new NotificationService();
  }

  placeOrder() {
    this.inventory.checkStock();
    this.payment.processPayment();
    this.shipping.createShipment();
    this.notification.sendEmail();
  }
}
```

---

### Client Code (Clean!)

```js
const order = new OrderFacade();
order.placeOrder();
```

✅ Client has **one method**
✅ Complexity hidden
✅ Subsystems can change freely

---

## 7. Key Characteristics

| Aspect           | Facade         |
| ---------------- | -------------- |
| Pattern Type     | Structural     |
| Goal             | Simplify usage |
| Coupling         | Reduced        |
| Subsystems       | Not modified   |
| Client Awareness | Minimal        |

---

## 8. Facade vs Adapter vs Bridge (Common Confusion)

| Pattern     | Purpose                                    |
| ----------- | ------------------------------------------ |
| **Facade**  | Simplify complex system                    |
| **Adapter** | Make incompatible interfaces work together |
| **Bridge**  | Separate abstraction from implementation   |

👉 Facade **does not change interfaces**
👉 Adapter **does change interface**

---

## 9. When Should You Use Facade? ✅

Use Facade when:

✔ Subsystem is complex
✔ Client code is cluttered with many calls
✔ You want a **clean API layer**
✔ You are designing **SDKs, libraries, or microservice clients**
✔ You want to protect clients from internal refactors

---

## 10. When NOT to Use Facade ❌

Avoid when:

* Subsystem is already simple
* Facade becomes a **God Object**
* You need full subsystem control at client level

---

## 11. Facade in Real Systems (Very Common!)

### Frontend

* API Service layer (`ApiClient`)
* UI talks to one service instead of many APIs

### Backend

* Microservice gateway
* BFF (Backend For Frontend)

### Enterprise Systems

* Banking APIs (very relevant to your HDFC context)
* Payment gateways
* CRM systems

---

## 12. Facade in System Design Thinking 🧠

Architects use Facade to:

* Create **layer boundaries**
* Reduce blast radius of changes
* Enable parallel team development
* Improve testability

> Facade = **API boundary**

---

## 13. Common Mistakes

❌ Putting business logic inside Facade
❌ Exposing subsystem objects through Facade
❌ One giant Facade for everything

✔ Facade should **coordinate**, not decide
✔ Keep multiple small facades if needed

---

## 14. Interview-Ready Definition 🎯

> “Facade is a structural design pattern that provides a simplified interface to a complex subsystem, reducing coupling and hiding internal implementation details.”

---

## 15. Quick Mental Model (Remember This)

> **Facade = One button to control many buttons**
